# Начало работы

## Описание проекта

Данный проект носит **образовательный** характер и представляет собой поэтапное создание ядра операционной системы для архитектуры **x86**. Каждая новая ветка будет содержать документ с подробным - или не очень 🙂 - описанием реализованных модулей ядра.

## Подготовка проекта

В файле [README.md](../../README.md) перечислены необходимые инструменты для создания, отладки и тестирования ядра ОС. Этот документ описывает процесс сборки, линковки и запуска проекта в эмуляторе [QEMU](https://www.qemu.org/).

## Как загружается операционная система

Перед началом стоит разобраться, как вообще происходит загрузка операционной системы. Сразу оговорюсь: объяснение будет довольно поверхностным, потому что часть процесса происходит «под капотом», а от нас многое не зависит - особенно с учётом того, что мы используем GRUB.

Итак, по порядку: этапы загрузки и причины выбора используемого инструментария.

Первое, что делает ПК после подачи питания, - выполняет аппаратную проверку POST (**Power-On Self-Test**). После завершения проверки встроенное ПО начинает искать загрузочное устройство: дискету, CD-ROM, жёсткий диск или другой носитель - в зависимости от настроенного порядка загрузки.

Затем BIOS проверяет диск на наличие загрузочной сигнатуры. Она представляет собой «магическое число», расположенное в последних двух байтах сектора.

В моём случае это `0x55` и `0xAA`:

```text
000001f0  00 00 00 00 00 00 00 00  00 00 00 00 00 00 55 aa  |..............U.|
```

После этого весь сектор размером 512 байт загружается в оперативную память по адресу `0x7C00`.

Что происходит дальше? Начинается выполнение кода, который только что был скопирован в оперативную память. В нашем случае это код загрузчика **GRUB**.

## Почему используется GRUB

При выборе инструментария я опирался на проблемы, описанные в разделе [Loading](https://web.archive.org/web/20241012032953/https://wiki.osdev.org/Boot_Sequence). Если честно, для меня это пока тёмный лес. Но главное, что я понял: GRUB действительно сильно упрощает жизнь.

Например, если загрузка происходит с жёсткого диска, где загрузочная запись занимает всего 446 байт, GRUB реализует **многостадийную загрузку**. Он самостоятельно включает защищённый режим, передаёт информацию о памяти и выполняет множество других подготовительных действий.

Короче, это натуральный **абсолютный кинотеатр**.

## GRUB

**GRUB** - это одна из первых программ, которая выполняется после включения компьютера. Как загрузчик, он отвечает за загрузку ядра операционной системы и передачу управления ему. Ядро, в свою очередь, инициализирует остальные компоненты ОС.

Конфигурация GRUB хранится в файле `grub.cfg`. В нём указывается, какое ядро и с какими параметрами нужно загрузить. В нашем случае используется стандарт **Multiboot2**, который позволяет гибко настраивать процесс загрузки: например, загружать ядро в графическом режиме или передавать ему дополнительные параметры.

Пример конфигурации:

```cfg
menuentry "JustOS" {
    multiboot2 /boot/JustOS-X86
    boot
}
```

Что это значит на практике:

* **`menuentry "JustOS"`** - создаёт пункт меню с названием `JustOS`.

* **`multiboot2 /boot/JustOS-X86`** - загружает ядро по стандарту Multiboot2.

* **`boot`** - передаёт управление загруженному ядру. Эта команда неявно выполняется в конце каждого `menuentry`, но здесь она указана явно для наглядности.

Когда я выбираю `JustOS` в меню GRUB, происходит следующее:

1. Выполняется команда `multiboot2`, которая загружает ядро в память.

2. Если загрузка прошла успешно, автоматически выполняется команда `boot`.

3. Управление передаётся в точку входа ядра.

## Скрипт компоновщика

Так как для загрузки ядра используется **Multiboot2/GRUB**, нужно обеспечить выполнение требований спецификации: загрузочный заголовок должен находиться в первых 32 КБ файла и иметь выравнивание 8 байт.

> An OS image must contain an additional header called _Multiboot2 header_, besides the headers of the format used by the OS image. The Multiboot2 header must be contained completely within the first 32768 bytes of the OS image, and must be 64-bit aligned. In general, it should come _as early as possible_, and may be embedded in the beginning of the text segment after the _real_ executable header.

Чтобы добиться нужного расположения секций, используется скрипт компоновки `linker.ld`.

```ld
ENTRY(_start)

SECTIONS {
    . = 1M;

    .text : ALIGN(4K) {
        . = ALIGN(8);
        KEEP(*(.multiboot))
        *(.text)
    }

    .rodata : ALIGN(4K) {
        *(.rodata)
    }

    .data : ALIGN(4K) {
        *(.data)
    }

    .bss : ALIGN(4K) {
        *(COMMON)
        *(.bss)
        *(.stack)
    }
}
```

> **Важно:** скрипт компоновщика нужен не только для корректного размещения Multiboot2-заголовка. Он также определяет, по каким адресам будут размещены секции ядра, и задаёт структуру итогового исполняемого файла.

Разберём конкретно заголовок и его расположение в исполняемом файле. Когда у линковщика есть набор [объектных файлов](https://habr.com/ru/articles/478124/), нужно гарантировать, что секция `.multiboot` окажется в начале итогового файла.

### Как скрипт размещает заголовок в начале файла

#### 1. Базовый адрес загрузки

```ld
. = 1M;  /* Ядро загружается по адресу 1 МБ в памяти */
```

#### 2. Принудительное размещение Multiboot2-заголовка

```ld
.text : ALIGN(4K) {
    . = ALIGN(8);           /* Выравнивание 8 байт для Multiboot2 */
    KEEP(*(.multiboot))     /* Разместить секцию первой и сохранить её */
    *(.text)                /* Остальной код располагается после заголовка */
}
```

#### 3. Механизмы обеспечения правильного порядка

* **`KEEP()`** предотвращает удаление секции линковщиком, даже если на неё нет явных ссылок.

* **Порядок перечисления секций** определяет порядок их размещения в итоговом файле.

* **`ALIGN(8)`** гарантирует 64-битное выравнивание, требуемое спецификацией Multiboot2.

#### 4. Результат в бинарном файле

Как показал эксперимент с `hexdump`, Multiboot2-заголовок располагается:

* по смещению `0x1000` от начала ELF-файла, после служебных заголовков;

* в первых 32 КБ файла, как требует спецификация;

* с корректным выравниванием 8 байт.

## Ассемблерный код загрузки ядра

Файл `boot.asm` содержит начальный код инициализации ядра. Он выполняет критически важные низкоуровневые операции подготовки системы перед передачей управления основной логике ядра на C++.

### Назначение и ключевые функции

* **Размещение Multiboot2-заголовка** - специальной структуры данных, которая идентифицирует ядро как совместимое со стандартом Multiboot2 и позволяет загрузчику корректно его загрузить.

* **Базовая инициализация процессора** - настройка критически важных регистров и режимов работы.

* **Подготовка стековой памяти** - создание корректного стекового кадра для последующего выполнения C++-кода.

* **Создание минимального окружения** - приведение системы в детерминированное состояние перед запуском высокоуровневого кода.

### Структура кода

```asm
section .multiboot
header_start:
    dd 0xE85250D6                ; Идентификатор Multiboot2
    dd 0                         ; Архитектура (0 = x86)
    dd header_end - header_start ; Длина заголовка
    dd 0x100000000 - (0xE85250D6 + 0 + (header_end - header_start)) ; Контрольная сумма
    dw 0                         ; Тип тега (0 = завершающий)
    dw 0                         ; Флаги тега
    dd 8                         ; Размер тега
header_end:

section .text
global _start
extern kernel_main

_start:
    cli                         ; Запрет аппаратных прерываний
    mov esp, stack_top          ; Инициализация указателя стека
    call kernel_main            ; Переход к основной функции ядра
    hlt                         ; Остановка процессора, защита от возврата

section .bss
stack_bottom:
    resb 16384                  ; Выделение 16 КБ для стека
stack_top:
```

### Принцип работы

После того как загрузчик GRUB обнаруживает и проверяет Multiboot2-заголовок, он загружает ядро в память по адресу 1 МБ, подготавливает базовое окружение и передаёт управление на метку `_start`.

Ассемблерный код выполняет минимально необходимую инициализацию:

1. отключает прерывания, чтобы гарантировать бесперебойную работу на ранних этапах;

2. настраивает стек для корректного вызова функций;

3. передаёт управление ядру, написанному на C++.

Стек размещается в отдельной секции `.bss`, которая автоматически обнуляется при загрузке.

## Настройка компилятора для встраиваемых систем

### Кросс-компиляция и кросс-компилятор

При разработке операционной системы нужно использовать **кросс-компиляцию**. Это процесс компиляции кода на одной платформе - например, на Linux - для выполнения на другой целевой платформе, в нашем случае на новой ОС для x86.

Обычный компилятор, встроенный в вашу систему, настроен для создания программ, которые работают в существующей ОС и используют её библиотеки и системные вызовы. Нам нужен **кросс-компилятор** (`i686-elf-gcc`), который генерирует код, не зависящий от конкретной операционной системы.

Это необходимо, потому что во время загрузки у нашей ОС ещё нет собственной стандартной библиотеки и среды выполнения.

### Отсутствие стандартной библиотеки

Обратите внимание на флаги компиляции `-nostdinc` и `-nostdlib`. Они указывают компилятору **не использовать стандартные библиотеки C и C++**.

Причина простая: этих библиотек не существует в среде нашей новой ОС. Функции вроде `printf`, `malloc` или `std::cout` зависят от сервисов хостовой ОС, например от системных вызовов Linux. В нашем ядре таких сервисов пока нет.

Поэтому всю необходимую функциональность - работу с памятью, вводом-выводом, строками и другими базовыми механизмами - придётся реализовывать самостоятельно :)

### Ключевые флаги компиляции в `toolchain-x86.cmake`

* **`-ffreestanding`** - указывает компилятору, что код будет работать в самостоятельной среде, а не в hosted-среде. Это отключает предположения компилятора о наличии стандартной библиотеки, но оставляет доступ к некоторым встроенным заголовкам, например `<stdint.h>` и `<stddef.h>`, которые нужны для низкоуровневого программирования.

* **`-m32`** - генерирует 32-битный код. Это важно, потому что в рамках этого руководства GRUB загружает ядро в 32-битном защищённом режиме.

* **`-nostdinc` / `-nostdlib`** - запрещают компилятору и компоновщику использовать стандартные системные заголовки и библиотеки.

* **`-fno-rtti`** - отключает RTTI (**Run-Time Type Information**) в C++. Для простого ядра этот механизм не нужен и только увеличивает размер бинарного файла.

* **`-mgeneral-regs-only`** - запрещает компилятору генерировать инструкции, использующие регистры SSE/MMX. Это упрощает обработку контекста при переключении задач и прерываниях, поскольку не требуется сохранять состояние этих регистров.

### Флаги компоновщика

* **`-T linker.ld`** - использовать пользовательский скрипт компоновки для точного контроля над размещением секций ядра в памяти.

* **`-m elf_i386`** - создать выходной файл в формате 32-битного ELF.

* **`--nmagic`** - отключить выравнивание секций по страницам памяти. Это позволяет плотнее упаковать ядро и уменьшить его размер.

Такой набор флагов и инструментов настраивает среду для создания чистого, независимого исполняемого файла ядра, который GRUB сможет корректно загрузить и которому сможет передать управление.
